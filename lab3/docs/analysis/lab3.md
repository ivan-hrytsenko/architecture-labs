# Аналітичний звіт: Впровадження патерну CQS (Лабораторна робота №2 vs Лабораторна робота №3)

## 1. Зміни в структурі проєкту порівняно з лабораторною роботою №2
Головною зміною у структурі проєкту став повний рефакторинг прикладного шару (`Application Layer`) та шару представлення (`Presentation Layer`) з метою розділення операцій запису та читання.

* **Application Layer (`src/application`)**: Замість універсальних сервісів або Use Cases, які відповідали за всі операції над сутністю (і читання, і запис), прикладний шар було чітко розділено на дві піддиректорії:
  * `commands/` — містить виключно класи-команди (DTO без логіки) та відповідні обробники (`Command Handlers`), які описують намір змінити стан системи.
  * `queries/` — містить класи-запити та їхні обробники (`Query Handlers`), які відповідають за вибірку та отримання даних без модифікації стану системи.
* **Presentation Layer (`src/presentation`)**: Контролери FastAPI стали максимально «тонкими». З них повністю зникла будьяка логіка оркестрації чи прямого виклику доменних методів. Тепер роль ендпоінту зводиться лише до приймання HTTP-запиту, його мапінгу в об'єкт `Command` або `Query` та передачі цього об'єкта відповідному обробнику.

---

## 2. Переваги CQS: чим допомогло розділення
* **Оптимізація шару читання**: Оскільки запити за своєю природою не змінюють стан системи, обробники запитів (`Query Handlers`) отримали можливість звертатися до бази даних напряму через SQLAlchemy, повністю минаючи доменні сутності, мапери та фабрики. Це значно знижує навантаження на CPU та пришвидшує генерацію відповідей.
* **Підвищення читаваності коду (Single Responsibility)**: Кожен Handler тепер є маленьким ізольованим класом, який вирішує рівно одне завдання (наприклад, тільки реєстрація користувача або тільки отримання списку товарів).
* **Полегшене тестування (Розділення стратегій)**: CQS дозволив чітко розділити тестування на дві швидкі та незалежні категорії:
  * Тести команд стали чистими Unit-тестами, де інфраструктура бази даних повністю замінюється на In-Memory заглушки (`fakes`). Вони перевіряють лише доменні інваріанти та бізнес-помилки, виконуючись миттєво.
  * Тести запитів стали інтеграційними, які фокусуються суто на правильності SQL-запитів, фільтрації та мапінгу полів, працюючи з реальною тестовою БД.

## 3. Недоліки CQS: що ускладнилося
* **Розрахунок кількості файлів (Class Explosion)**: Замість одного файлу сервісу, який містив усі методи для сутності, тепер на кожну окрему операцію створюється мінімум два класи (`Command/Query` + `Handler`), що призводить до значного збільшення кількості дрібних файлів у проєкті.
* **Шаблонний код для мапінгу**: Контролери змушені виконувати додатковий крок трансформації: спочатку валідувати вхідний HTTP-запит через Pydantic-схему, потім трансформувати її в об'єкт `Command` або `Query`, і лише після цього передавати в Handler.
* **Ускладнення DI-контейнера**: Файл `dependencies.py` став значно більшим і складнішим, оскільки тепер для кожної команди та кожного запиту необхідно явно конфігурувати окрему фабрику для збирання обробника та ін'єкції репозиторіїв чи сесій бази даних.

## 4. Відмінність Command/Query Handler від універсального Service та вплив на розширюваність
Універсальний сервіс (наприклад, `ProductService`) накопичує в собі всі можливі операції над сутністю. З часом, коли додаються нові бізнес-вимоги, такий сервіс перетворюється на «божественний об'єкт» (God Object) на тисячі рядків коду. Зміна одного методу в такому сервісі підвищує ризик випадково зламати сусідні методи, оскільки вони мають спільні залежності всередині класу.

Підхід із **Command/Query Handlers** кардинально змінює розширюваність системи:
* **Додавання нової фічі**: Якщо в систему потрібно додати нову команду (наприклад, `ChangeProductStatus`), ми просто створюємо новий ізольований файл із цією командою та її обробником. Ми взагалі **не чіпаємо** існуючий працездатний код створення чи видалення продуктів, що зводить ризик регресійних помилок до нуля.
* **Принцип відкритості/закритості (OCP)**: Система стає відкритою для розширення (додавання нових обробників), але закритою для модифікації (старі обробники залишаються недоторканими).

## 5. Обґрунтування використання Read Models (DTO) на рівні запитів
У лабораторній роботі №2 операції читання повертали повноцінні насичені доменні моделі (`User`, `Product`), які містили в собі бізнес-поведінку та правила валідації інваріантів.

У CQS запити повертають спеціальні **Read Models** (DTO), повністю відірвані від домену.

### Чому це важливо?
1. **Асиметрія даних**: Те, що клієнту потрібно побачити на екрані (наприклад, список товарів із виведеним іменем фермера та рейтингом), часто кардинально відрізняється від того, як дані зберігаються в системі та як захищаються доменні інваріанти. Read Models дозволяють легко проектувати плоскі структури (плоскі DTO), оптимізовані под конкретні потреби інтерфейсу.
2. **Продуктивність**: Нам не потрібно витрачати ресурси процесора на ініціалізацію складних об'єктів домену (таких як Value Objects `Price` або `Quantity`) під час звичайного виведення списку з сотень товарів. Ми читаємо "сирі" дані з бази даних і відразу пакуємо їх у швидкі легковажні структури.
3. **Безпека та ізоляція**: Використання Read Models гарантує, що клієнт через API отримає лише ті поля, які йому дозволено бачити, виключаючи випадковий витік внутрішнього стану доменних моделей назовні.
