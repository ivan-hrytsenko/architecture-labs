# Аналітичний звіт: Порівняння архітектурних підходів (Лабораторна робота №1 vs Лабораторна робота №2)

## 1. Зміни в структурі проєкту порівняно з лабораторною роботою №1
У лабораторній роботі №1 використовувався класичний "Transaction Script" (процедурний) підхід, де структура каталогів групувалася за технічними ознаками файлів (`models`, `services`, `schemas`). Бізнес-логіка була розмита між контролерами FastAPI та функціями сервісного шару, які безпосередньо залежали від об'єктів SQLAlchemy ORM.

У лабораторній роботі №2 проєкт перейшов на чотиришарову Чисту архітектуру (Clean Architecture) з суворим правилом залежностей (Dependency Rule), спрямованим всередину ядра:
* **Presentation Layer (`src/presentation`)** — взаємодіє із зовнішнім світом через HTTP-протокол. Містить лише DTO (Pydantic), ендпоінти FastAPI та механізми автентифікації. Залежить виключно від прикладного шару.
* **Application Layer (`src/application`)** — координує виконання бізнес-сценаріїв (Use Cases). Він не містить прямої логіки перевірки даних, а лише керує потоком даних між інтерфейсами репозиторіїв та доменними об'єктами.
* **Infrastructure Layer (`src/infrastructure`)** — містить технічні деталі реалізації: конфігурацію бази даних, ORM-моделі SQLAlchemy, конкретні класи репозиторіїв та мапери.
* **Domain Layer (`src/domain`)** — ізольоване ядро системи. Складається з чистих Python-класів (Entities, Value Objects), фабрик (Domain Factory) та абстрактних інтерфейсів репозиторіїв. Цей шар не має жодних зовнішніх залежностей (включаючи SQLAlchemy чи Pydantic).

## 2. Переваги розділення на шари
Перехід до шарової архітектури дозволив вирішити декілька фундаментальних проблем першої ітерації проєкту:
* **Чіткий поділ відповідальності (Separation of Concerns):** Кожен клас має одну зону відповідальності. FastAPI займається виключно серіалізацією/десеріалізацією JSON та HTTP-статусами, Use Case — оркестрацією, Домен — бізнес-правилами, а SQLAlchemy — генерацією SQL-запитів.
* **Ізоляція бізнес-логіки від фреймворків:** Будь-які зміни в синтаксисі SQLAlchemy або оновлення FastAPI до нових версій жодним чином не впливають на ядро системи, де закодовані правила маркетплейсу.
* **Тестова придатність (Testability):** Завдяки принципу інверсії залежностей (DIP), тестування бізнес-правил більше не потребує підняття реальної бази даних, Docker-контейнерів чи створення файлів SQLite на диску. Ми тестуємо чистий Python-код за допомогою In-Memory заглушок репозиторіїв, що пришвидшило виконання юніт-тестів у десятки разів.

## 3. Недоліки та ускладнення, що з'явилися
За підвищення гнучкості та чистоти коду довелося заплатити ускладненням кодової бази:
* **Збільшення обсягу шаблонного коду (Boilerplate):** Для додавання однієї сутності тепер необхідно створити доменну модель, інтерфейс репозиторію, ORM-модель, мапер, принаймні один Use Case, DTO-схеми та конкретну реалізацію репозиторію.
* **Необхідність явного мапінгу:** Оскільки доменні моделі ізольовані від ORM, з'явився додатковий шар `DataMapper`. Постійна конвертація об'єктів з `UserORM` у `User` і назад створює додаткове навантаження на CPU та вимагає написання додаткового коду трансформації полів.
* **Складніша навігація по проєкту (Cognitive Load):** Слідкувати за потоком виконання програми стало важче через глибокий стек викликів та високий уровень абстракції, що вимагає більшої концентрації під час розробки та відладки коду.

## 4. Гнучкість системи при зміні бази даних або веб-фреймворку
Завдяки застосуванню патерну "Репозиторій" та Dependency Inversion Principle, гнучкість системи зросла до максимуму:
* **Зміна бази даних/ORM:** Якщо виникне потреба перейти з SQLAlchemy на TortoiseORM або взагалі на NoSQL базу даних (наприклад, MongoDB), прикладний шар та домен залишаться абсолютно незмінними. Нам достатньо буде створити новий клас інфраструктури (наприклад, `MongoUserRepository`), який реалізує абстрактний інтерфейс `UserRepository`, та підставити його у конструктор Use Case через Dependency Injection.
* **Зміна веб-фреймворку:** Якщо завтра FastAPI замінить інший асинхронний фреймворк (наприклад, Litestar або Sanic), зміни зачеплять виключно файли шару представлення (`src/presentation/routers.py`). Уся логіка реєстрації користувачів та створення продуктів, зашита в Use Cases, продовжить працювати без модифікацій.

## 5. Обґрунтування вибору моделі предметної області (Rich Domain Model)
Для лабораторної роботи №2 було обрано **Насичену доменну модель (Rich Domain Model)** замість Анемічної (Anemic).

**Причини вибору:**
1. **Захист інваріантів у місці виникнення:** Бізнес-правила (валідність електронної пошти, обмеження ролей, невід'ємність ціни продукту або зменшення залишків на складі) інкапсульовані безпосередньо всередині сутностей `User` та `Product` та їхніх `Value Objects`. Об'єкт неможливо створити або перевести в некоректний стан, оскільки перевірка відбувається автоматично в конструкторі або методах сутності.
2. **Запобігання розмиттю логіки:** В анемічній моделі сутності є просто структурами даних (набором полів), а логіка розпорошується по всьому проєкту. Насичена модель концентрує поведінку в одному місці, роблячи код самодокументованим та надійним.
3. **Контроль стану через фабрики:** Використання `DomainFactory` гарантує єдину точку входу для створення об'єктів предметної області, що усува дублювання коду валідації в різних Use Cases та захищає систему від появи аномальних даних.
