# Farm Products Marketplace System

## Опис проєкту
Система автоматизації онлайн-маркетплейсу для фермерських продуктів. Платформа підтримує реєстрацію користувачів з різними ролями (фермери та покупці), публікацію товарів за категоріями, оформлення замовлень із автоматичним контролем залишків на складах та систему відгуків.

## Технологічний стек
* **Мова програмування:** Python 3.13
* **Веб-фреймворк:** FastAPI (ASGI)
* **База даних:** PostgreSQL / SQLite (для тестів)
* **ORM:** SQLAlchemy 2.0
* **Токени безпеки:** bcrypt, python-jose
* **Тестування:** pytest 9.0

## Базове налаштування проєкту
Ці кроки виконуються один раз для всього циклу розробки.

```bash
git clone https://github.com/ivan-hrytsenko/architecture-labs.git
python -m venv .venv

pip install -r ./lab1/requirements.txt

```

Перша ітерація системи, написана у форматі швидкого базового прототипу (анемічна доменна модель, бізнес-логіка розмита між контролерами та статичними сервісами).

* `lab1/tests/` — Інтеграційні тести API

#### Запуск бази даних (Docker)

```bash
docker run --name farm-postgres -e POSTGRES_USER=user -e POSTGRES_PASSWORD=password -e POSTGRES_DB=farm_products -p 5432:5432 -d postgres

```

Створіть файл `.env` у папці `lab1/`:

**Варіант А: Локальна SQLite (Швидкий старт без Docker)**

**Варіант Б: PostgreSQL у Docker (Аналог production-середовища)**

### Лабораторна робота №4: Синхронна та асинхронна комунікація між компонентами

Рефакторинг системи з метою виділення побічних операцій (надсилання сповіщень) в окремий ізольований компонент. Реалізовано та протестовано два варіанти взаємодії між модулями: синхронний (прямий виклик через інтерфейс із забезпеченням атомарності) та асинхронний (подієво-орієнтований через In-Process Event Bus).

#### Структура модулів

* `lab4/src/domain/` — Ядро системи: насичені моделі (`User`, `Product`), Value Objects, фабрика, інтерфейси та інтеграційні події.
* `lab4/src/application/commands/` — Обробники команд (`Command Handlers`), адаптовані для роботи в синхронному (із відкотом стану при збоях) та асинхронному режимах.
* `lab4/src/application/common/` — Інфраструктура шини подій (`EventBus`) для реєстрації підписників та публікації повідомлень.
* `lab4/src/notifications/` — Ізольований допоміжний компонент (інтерфейс, сервіс та консюмер) для обробки побічних операцій надсилання сповіщень.
* `lab4/src/infrastructure/` — Шар інфраструктури: ORM-моделі, мапери даних (`DataMapper`) та репозиторії (додано метод `delete` для забезпечення консистентності даних).
* `lab4/src/presentation/` — Шар представлення: ендпоінти FastAPI, схеми валідації Pydantic та налаштування залежностей.
* `lab4/tests/` — Розділене інтеграційне тестування:
  * `tests/sync/` — Перевірка поведінки системи при збоях зовнішніх сервісів та відкоту бізнес-операцій.
  * `tests/async/` — Перевірка неблокуючої доставки подій та досягнення кінцевої узгодженості (Eventual Consistency).

### Лабораторна робота №5: Модульний моноліт та Шар запобігання корупції (Anti-Corruption Layer)

Рефакторинг системи із переходом від традиційного шаруватого моноліту до архітектури **модульного моноліту (Modular Monolith)**. Систему декомпоновано на автономні обмежені контексти (Bounded Contexts) із повною ізоляцією моделей даних, а міжмодульну взаємодію реалізовано через шину подій та шар захисту домену від деградації.

#### Структура модулів

* `lab5/src/common/` — Спільне ядро (**Shared Kernel**): базова конфігурація бази даних, спільні класи інтеграційних подій та шина `EventBus`. Не залежить від бізнес-модулів.
* `lab5/src/core/` — Основний контекст (**Core Context**): бізнес-вертикаль маркетплейсу (користувачі, продукти), що володіє власними ізольованими ORM-моделями (`UserORM`, `ProductORM`) та генерує події змін.
* `lab5/src/analytics/` — Контекст аналітики (**Analytics Context**): ізольований допоміжний модуль, що збирає метрики маркетплейсу та будує власні плоскі проекції даних (`UserStatsORM`, `ProductMetricsORM`).
* `lab5/src/analytics/acl.py` — Шар запобігання корупції (**Anti-Corruption Layer — ACL**): перехоплює зовнішні події з Core-контексту, трансформує їх у внутрішній аналітичний формат та забезпечує кінцеву узгодженість даних (Eventual Consistency).
* `lab5/src/presentation/` — Шар представлення: тонкі ендпоінти FastAPI та глобальне налаштування залежностей системи.
* `lab5/tests/` — Наскрізне інтеграційне тестування взаємодії модулів із застосуванням **`StaticPool`** у фікстурах для коректної ізоляції транзакцій SQLite в пам'яті.
